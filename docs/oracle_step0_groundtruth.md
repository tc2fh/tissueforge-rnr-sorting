## 1. What the oracle is (corroborated)

`3DVertVor/` = the Manning-lab fork of **`tvm`** (Zhang & Schwarz, PRR 4:043148), a **C++** 3D
vertex model. It adds multiple cell types + **heterotypic interfacial tension** (`Energy/
Interface.{h,cpp}`) + `.vtu` output. Layout: `Cell/ Edge/ Energy/ Polygon/ Reconnection/ Run/
Vertex/ misc/ cmake/ scripts/`, plus top-level `tvm.cpp`, `main.py`, `CMakeLists.txt`, `README.md`.
Has an explicit **`Edge`** class (we don't) and `Reconnection/Reconnection.cpp` (`I_H`/`H_I`).

## 2. How it is configured + run

Two input files, both read at startup (`tvm.cpp::InitializeAll` + `LoadConf`):

1. **`sample.topo`** — the initial mesh topology (vertices / edges / polygons / cells lists).
   Generated by **`main.py`** (top-level), which uses **`pyvoro`** to build a **periodic** Voronoi
   tessellation. Defaults: box **`Lx=Ly=Lz=12`**, **`Nvertices = Lx·Ly·Lz = 1728` seeds → 1728
   cells**, `periodic=[True,True,True]`. It asserts every vertex has **exactly 4 neighbouring
   cells** (the 4-valent assumption the Okuda I↔H needs). Writes `sample.topo`.
   - (`scripts/tvm/main.py` is a near-duplicate generator; `scripts/vtk/main.py` is a `.vtu`
     post-processor. `README.txt` in `scripts/tvm` is 2 lines, no usage beyond "run main.py".)
2. **`conf`** — run parameters, one per line (`tvm.cpp::LoadConf`, lines ~288-665). Keys:
   - `time t_start t_end dt`  → also sets `dtr_ = 10·dt` (reconnection cadence).
   - `dump vtk <period>`      → `.vtu`/dump cadence (simulation-time units).
   - `log <period>`           → screen-log cadence.
   - `s01,s04,s05,s06,s07`    → per-type **target surface area** s₀ (s02,s03 hard-coded 6.70).
   - `v01,v04,v05,v06,v07`    → per-type **target volume** v₀ (v02,v03 hard-coded 1.3).
   - `l0s`                    → a spring rest-length (the `SpringGon` machinery; unused for us).
   - `Lth [verbose]`          → **reconnection threshold** (Condition 2 edge length).
   - `T`, `Tbot`              → **temperature** = active self-propulsion strength (NOT thermal kT
                                in the usual sense — drives a per-cell motility director; §5).
   - `sigma12 sigma14 sigma15 sigma45 sigma46 sigma56 sigma17 sigma47 sigma57 sigma67` →
     **heterotypic interfacial tensions by type pair** (others default in `Run.h`, e.g.
     `sigma23=0.5, sigma31=3, sigma36=2`). These are the Manning σ_ij — our `Adhesion` analogue.
   - Run is invoked simply as **`./tvm`** in a directory containing `sample.topo` + `conf`.

**Driver schedule is hard-coded for a delamination/stratification experiment, NOT clean
sorting** (`Run::start`, lines 448-468 + `InitializeCellUniform/Type/Delam`):
- `InitializeCellUniform()` sets **every** cell to `type_=1, s0=5.6, v0=1.0` (single type).
- At sim-time **t=25** → `InitializeCellType()`; with its default switches (`topbot=1`,
  `randompos=0`) this assigns types **by z-height** (bottom half → type 0, top half → type 7) —
  i.e. it **starts already top/bottom-segregated**, not salt-and-pepper.
- t=75 → `temperature_=0`. t=200 → `InitializeCellDelam()` (a third "delaminating" type).
- So **to run OUR regime** (random 50/50, γ_homo=0/γ_het=1, passive) we must edit the driver:
  use `randompos=1` in `InitializeCellType` (random 50/50), set the two cell types to **{0,1}**
  (so the het interface polygon-type = 0+1 = **1** → controlled by `sigma12`; homotypic 0+0=0 and
  1+1=2 match no σ key → tension 0), set `sigma12=1` and all other σ=0, set `temperature=0`
  (passive), and drop the t=200 delam call. These are local edits to GPL code **run** as an
  oracle (never copied into `rnr/`).

## 3. What it outputs (and the dump cadence)

Per `dump_period` (`Run.cpp:1025-1037`, only after the VTK/log block):
- **`paraview/cells_t0_<time>.vtu`** — binary `vtkUnstructuredGrid`; per-polygon arrays
  `polyID, polyType, cellID, cellType, cellCenter, cellVol, PolyNeighs` (`Run.cpp:1540+`).
  `cellType` per cell + `PolyNeighs` (the two cells touching each polygon) is enough to recompute
  het-fraction(t) ourselves if we go the `.vtu` route.
- **`topo.txt`, `cellcenter*.txt`, `cellShapeIndex*.txt`, `reconnection*.txt`** — plain-text, no
  VTK needed.
Per `log_period`, the **screen log** prints columns:
`Time  Rte  Volume  I->H  H->I  E_volume  E_interface  Energy  AvgForce  MaxForce`
(`Run.cpp:402-526`). `I->H`/`H->I` are reconnection counts since the last log; `E_interface`
includes the het-tension energy. **`dumpDemix()` (→ `demixing.txt`) is NOT called by default** —
adding the call is the cheapest way to get the metric directly.

## 4. The demixing metric — same as ours (re-derived from `dumpDemix`, `Run.cpp:1751`)

For each cell: `hom_frac = (#faces that are NOT the heterotypic pair) / (#faces)`, then
```
demix_oracle = mean_over_cells( 2·(hom_frac − 1/2) )            (count/face-based, ∈ [−1,1])
```
With a 2-type {0,1} system the heterotypic face has polygon-type 1, so "NOT type 1" = homotypic.
This is **exactly the negative of our `metrics.demixing_index`**:
`D_ours = mean(2·(het_frac − ½))` and `het_frac = 1 − hom_frac`, so **`demix_oracle = −D_ours`**.
(Confirms our metric's flagged sign convention: we sign it so sorted = more negative; the oracle
signs it so sorted = more positive.) Caveat: oracle weights each **face** equally; our `on_D`
weights each **neighbour cell** equally — nearly identical for foam-like cells, a minor caveat.
Both are **count-based**, NOT area-weighted (so compare against our `on_D`, with our area-weighted
`on_af` shown as a secondary curve).

## 5. The exact energy form (re-derived; cite, never copied)

From `Energy/Volume.cpp` + `Energy/Interface.cpp` (`updateEnergy`):
```
E_total = Σ_cells [ kv·(V − v0)²  +  (s − s0)²  +  (1/2)·Σ_{faces f of cell} σ(type_f)·A_f ]
```
- **Volume:** `kv=10` (`Volume.cpp:23`), `v0` per-type (1.0 uniform). Pressure `= −2·kv·(V−v0)`.
  → our **`VolumeConstraint`**. Note V is a **signed** volume (`updatePolygonDirections`/
  `correctDirection`) — the same signed-volume convention behind our winding-sign-flip finding.
- **Surface elasticity:** `(s − s0)²` per cell, `s0≈5.6` — a **quadratic shape/area-elastic
  term** (the polygon tension contribution is its derivative `2·(s − s0)`, `Interface.cpp:179`).
  This is the **SAME functional form** as our TF `SurfaceAreaConstraint`, `λ_S(A−A0)²`
  (tfSurfaceAreaConstraint.cpp:34) — verified post-build; they differ only in coefficient
  (oracle 1 vs ours 0.1) and target shape index (5.6 vs 6.0). [Corrects an earlier draft that
  called this a form difference.]
- **Heterotypic interfacial tension (the sorting driver):** `(1/2)·Σ σ_ij·A_ij` summed over both
  cells of each interface = `Σ_interfaces σ_ij·A_ij` — **AREA-based, linear in interface area**,
  by cell-type pair. NOTE (corrects an earlier draft): our TF `Adhesion` is the σ_ij analogue but
  is **EDGE/LINE-based**, not area — `λ·Σ(edge length × #partner-type neighbour surfaces)`
  (tfAdhesion.cpp:137). So this term is the **genuine functional difference** between the codes;
  `metrics.contact_summary`'s `Σλ·A` is an area *proxy*, not the actor energy. For our regime:
  `σ_homo=0, σ_het=1`.
- **Integrator** (`updateVerticesVelocity/Position`): overdamped `v = μ·(F_vol+F_int)`, `μ=1`,
  drift (centre-of-mass velocity) removed, then `x += v·dt + dt·motility`. **`motility` is active
  self-propulsion** from a per-cell director, magnitude set by `temperature_` (`updateSP`). Set
  `temperature=0` to match our passive sorting. Reconnection (Okuda I↔H) every `dtr=10·dt`.

**Energetics verdict (corrected post-build):** **volume + surface terms are the SAME quadratic
Hookean form** on both sides (NOT a form difference, as an earlier draft claimed). The **genuine
functional difference is the het tension**: oracle = **area** `σ·A`; ours (TF `Adhesion`) =
**edge/line** tension. Coefficients/scale also differ (surface stiffness 1 vs 0.1, volume 10 vs 1,
cell V0 1 vs 4, shape idx 5.6 vs 6.0), so the oracle's het-to-surface drive ratio is ~1/10 of
ours → **σ=1 is NOT a matched drive**. The oracle also has optional active motility (off when
`temperature=0`). So compare the **demixing trend/shape** only — never absolute numbers (their box
is periodic; ours finite; the energetics are not identical).

## 6. Build feasibility on this machine (the decision gate)

Tooling present: `cmake` (Homebrew), `clang++`/`g++` (Apple), `brew`; **`eigen` is already a pixi
dep**. **No VTK, no conda, no python2.7.**

The `3DVertVor` `CMakeLists.txt` `find_package`s **Eigen3 (have it) + VTK (don't) + NetCDF
(optional)**, and `tvm.cpp` `#include`s **`python2.7/Python.h`** + links **`-lstdc++fs`**
(`std::experimental::filesystem`). On macOS Darwin 25 these are three real friction points:
VTK ≥9 also dropped the legacy `${VTK_USE_FILE}` / component names the CMake uses.

**BUT a VTK-free build path exists** (this is the key Step-0 finding):
- VTK is used **only** inside `dumpConfigurationVtk()` (the `.vtu` writer), which is gated by the
  runtime flag `bool outputVtkVis = true` (`Run.cpp:420`) — but it still must *link*. We don't
  need `.vtu` at all: the **demixing metric + topo dumps are plain-text**, and the screen log
  already prints `E_interface`, `I->H`, `H->I`. So we can `#ifdef`/strip the VTK include + the
  `dumpConfigurationVtk` body, drop the VTK `find_package`, and build with **just C++11 + Eigen**.
- `python2.7/Python.h` is included only for the **commented-out** `createfile()` → strip the
  include.
- `std::experimental::filesystem` (the `paraview/` mkdir) → guard out with the VTK block.
- The base **`tvm/CMakeLists.txt` needs NONE of these** (pure C++11, writes `.vtk` as text) —
  proof the engine compiles without VTK; only the fork's `.vtu` path added the dep.

**Estimated effort:** ~30–60 min — strip 3 deps from a *local copy* of `3DVertVor` source +
reconfigure the driver for our regime (§2) + generate `sample.topo` (pyvoro is in our env) + write
a `conf` + add the `dumpDemix()` call. All edits are to GPL code we **run** as an oracle; **nothing
is copied into `rnr/`**. Risk: the driver/reconnection has experiment-specific cruft (springs,
delamination) that may need more stripping; time-box it and fall back to a literature-anchored
reference if it fights past ~1h.

**Decision:** feasible enough to attempt a **bounded** no-VTK build (the plan's Step 2), since no
sample trajectory exists and that is the only path to a real-data overlay. Flag to the user first
per the guardrail ("ask before a painful GPL/VTK build"), recommending the bounded attempt.

**OUTCOME (2026-05-30): the bounded build SUCCEEDED.** User approved; the VTK-free build works
(see `oracle_run/README.md` for the exact patches). Extra hurdles beyond §6's prediction: (a) bump
C++11→17 (modern conda Eigen needs ≥14); (b) 3 missing-`return` UB functions trap under clang19
`-O3`; (c) **pyvoro-mmalahe's `periodic=True` is broken** → periodicity generated with scipy
instead (`oracle_run/make_periodic_topo.py`); (d) the box size is **hard-coded** `Lx_=12` in
`Run::Run()` and must match the generated topo box (6³) or periodic volumes blow up. The run
produces a real `demixing.txt`; the comparison verdict is in `rnr_sorting_notes.md §8` (short
version: periodic bulk does NOT break our ceiling at matched σ=1; the limit is largely physical).
